# Blansole backend work log — 2026-07-21

## Scope

ตรวจรายงาน Frontend ↔ Backend data flow เทียบกับโค้ดปัจจุบัน แล้วเติมเฉพาะส่วนที่ยังขาดหรือเป็น stub โดยไม่สร้างผลทางการแพทย์, BLE protocol หรือ provider integration ที่ไม่มี specification/credential จริง

## API และข้อมูลจาก frontend

- รองรับ `birthDate` เป็น alias ของ `birthday`, รับ `age` เพื่อ compatibility โดยใช้วันเกิดเป็น source of truth และรองรับการสร้าง `goal` ระหว่างอัปเดต profile
- รองรับ NDJSON timestamp field `t` ทั้ง Unix seconds และ milliseconds
- คง authentication บน session sync endpoint และจำกัด batch ที่ 20,000 samples/5 MB
- เพิ่มเอกสาร contract สำหรับ health integration, notification devices, admin content และ program recommendation

## Health integration

- เพิ่ม connect/list/disconnect สำหรับ Apple Health และ Health Connect
- รับ daily summaries ที่อ่านจากโทรศัพท์ แล้วส่งเข้า BullMQ แทนการพยายามอ่าน Apple Health จาก server
- เพิ่ม health-sync worker สำหรับ validate ownership, upsert daily summary, ปิด sync log และสร้าง notification event
- จำกัดหนึ่งคำขอไม่เกิน 366 daily summaries และตรวจช่วงค่าของ steps, distance, calories และ heart rate

## Session processing และ performance

- คำนวณ steps, distance, calories, duration, cadence, speed, pace และ step length จากข้อมูลที่วัดได้
- สรุป relative forefoot/midfoot/heel pressure, hotspot และ balance โดยใช้ `pressureLevel=unclassified`
- บันทึก measured gait values เฉพาะค่าที่มีอยู่จริง และระบุ algorithm version ทุกชุด
- ไม่สร้าง clinical risk, fall risk หรือ pain score จาก heuristic ที่ยังไม่ validate
- เพิ่ม regression test สำหรับ batch ขนาด 20,000 samples พร้อม worker budget 5 วินาที

## Admin content และ recommendation

- เพิ่ม program/video/tag/rule CRUD ที่จำเป็น พร้อม submit-review, publish และ unpublish
- จำกัด publish/unpublish ให้ Admin และบังคับ privileged role ผ่าน 2FA
- ตรวจ YouTube URL เฉพาะ HTTPS host ที่อนุญาต, canonicalize video ID และเรียก fixed oEmbed endpoint เพื่อป้องกัน SSRF
- ไม่ archive published video เมื่อ YouTube/network เกิด transient error; archive เฉพาะสถานะ broken/private/removed ที่ยืนยันได้
- เพิ่ม audit/review logs และกำหนดให้วิดีโอทุกตัวมี AI grounding description
- เพิ่ม deterministic recommendation ที่เลือกเฉพาะ published program/video จาก database rules และคืน `matched=false` แทนการสร้าง content ขึ้นเอง

## Notification และ scheduled workers

- เพิ่ม notification outbox processing, idempotency จาก `sourceEventId`, rule conditions และ cooldown
- เพิ่ม in-app delivery และ push device registration API โดยไม่ส่ง token เต็มกลับ client
- เพิ่ม Expo Push adapter; สถานะจาก send API บันทึกเป็น `accepted` จนกว่าจะมี receipt confirmation
- แก้ preference precedence ให้ category-specific setting override ค่า `all`
- direct FCM/APNs ถูกบันทึกเป็น `not_configured` จนกว่าจะมี adapter และ credential rotation procedure
- เพิ่ม scheduler สำหรับ notification sweep, YouTube link checks และ weekly/monthly episode summaries
- เพิ่ม deterministic episode aggregate พร้อม unique period key เพื่อให้ retry ได้โดยไม่สร้างข้อมูลซ้ำ

## AI safety

- AI session summary อ่าน metrics, pressure, gait และ validated risk records จาก database ก่อนเรียก model
- system prompt ห้ามสร้าง metric ที่ไม่มี, ห้ามวินิจฉัย และห้ามแปลง `unclassified` pressure เป็น medical risk
- internal FastAPI queue endpoint ต้องใช้ `INTERNAL_API_KEY` และ compare แบบ constant-time
- session summary upsert ตาม `sessionId` และใช้ prompt version `grounded-session-v2`

## Authentication และ production security

- production startup ปฏิเสธ default/short `JWT_ACCESS_SECRET`
- เข้ารหัส TOTP seed ด้วย AES-256-GCM และแยก `TOTP_ENCRYPTION_KEY`
- รองรับการอ่าน TOTP plaintext เดิมระหว่าง migration
- token หลัง OAuth ของ Admin/Content Editor ต้อง verify 2FA ก่อนใช้ privileged route
- เพิ่ม tests สำหรับ production JWT fail-fast, TOTP encryption/decryption และ missing-key fail-closed

## Database และ migration

- เพิ่ม push registrations, notification processing state/source event relation และ episode summary unique period key
- แก้ migration เดิมที่สร้าง `session_sensor_samples` เป็น UUID ทั้งที่ primary/foreign IDs ของ schema เป็น TEXT
- แก้ migration ใหม่ให้ `source_event_id`, push registration ID และ user ID ใช้ TEXT ตรงกับ Prisma schema
- ทดสอบ fresh PostgreSQL/pgvector จริง: migration ทั้ง 5 ชุดผ่านครบ ใช้เวลา 3.84 วินาที
